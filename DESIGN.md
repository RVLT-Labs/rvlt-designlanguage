# RVLT Flow — Design Language &amp; Brand

This is the brand. RVLT Flow is **production operations software, built by a
production company** — and the design has to feel that way: made by a live-events
crew, not generated by a SaaS template.

**Canonical reference implementation:** [`index.html`](index.html) — dark
"espresso", spotlit. Everything below is pulled from it. Light-theme twin (same
system, light surface): [`light.html`](light.html).

> **North star.** Personality is the anti-slop weapon. Left unconstrained, AI
> averages toward purple-gradient + Inter + three icon-cards. We win by importing
> a *real world* — road cases, gaff tape, flight labels, dispatch sheets, crew
> culture — and by writing like someone who's actually done a load-out.

**Two surfaces, one dial — Expressive ⟷ Functional.** Same brand, red, type and
voice at different volume:
1. **Marketing** (website) — expressive: gear stickers, big display type,
   hand-drawn doodles, wit, the spotlight pops.
2. **Product** (app — Next.js 16 · Tailwind v4 · shadcn/ui) — same tokens and
   personality, dialled down for data density: fewer stickers, calmer motion,
   tighter spacing, a data mono.

**The app is dark-first**, so the brand's primary surface is the **warm espresso
dark** (§3.1). Light "Paper" (§3.2) is the alternate.

## 0. Using this document — AI build guide

**You are an AI coding agent (e.g. Claude Code) building the RVLT Flow _app_ from
this spec. It is authoritative. Follow these rules:**

- **MUST use only tokens/values defined here** (§3 colour, §5 type, §6–7
  spacing/shape, Appendix A). Do **not** invent hex codes, off-scale font sizes,
  or new radii. If you need something that isn't here, derive it from the nearest
  token and leave a `/* TODO: confirm with brand */` note — never start a parallel
  system.
- **Default theme = dark "espresso"** (§3.1): `:root` is dark, `.light` is the
  opt-in alternate. The product is dark-first.
- **Build at the _Functional_ end of the dial (§15) — but the app must still be
  FUN.** The rule is **"calm data, fun chrome"** (§15.5): dense tables stay clean
  and legible; personality + colour go into the chrome &amp; moments — sidebar,
  modals, toasts, empty states, success animations, the mascot. Restyle shadcn
  primitives with these tokens; Lucide icons; data mono for figures; keep button
  press feedback; drop scroll parallax. **The signature product screens are
  specified in §15.4 — build the app to look like those.**
- **Stack:** Next.js 16 · Tailwind v4 · shadcn/ui. Paste §15.1 into
  `app/globals.css`; use the recipes in §15.2.
- **Precedence on conflict:** paste-ready code (§15.1, Appendix A) > token tables
  (§3/§5/§7) > prose. The reference mockup `index.html`
  demonstrates the _expressive_ end — copy its tokens and feel, **not** its
  sticker/doodle density, into the app.
- **Self-audit every screen/component before returning** (§15.3 for app, §12 ship
  checklist for marketing).
- **Escalate to the human only for:** the BC Alphapipe Typekit URL, real
  data/content, or brand calls not covered here. Otherwise proceed with the
  documented defaults — don't stall.
- **Section applicability:** §§2, 4, 9–11, 14 lean **marketing**; §§3, 5–8, 12–13,
  15 + Appendix A apply directly to **app code**. When unsure for the app, prefer
  the Functional guidance.

## 2. Voice &amp; language

The single most important brand asset after the red. Talk like a production
manager, not a marketer.

**Positioning (confirmed).** "Built for production companies, by a production
company." Flow wasn't built to sell software — it was built inside Two Toned
Productions to fix the team's own ops problems, and used on real jobs from day
one. It's shaped by real production pain: gear clashes, late quote changes, crew
gaps, missing accessories, damaged returns, compliance checks, venue quirks, and
knowledge that lived in people's heads. Lead with the origin, not features.

**Rules**
- **Name the real job/pain**, not an aspiration.
  - ✅ "Flow wasn't built to sell software. It was built to solve problems."
  - ✅ "Stop promising gear you've already booked out."
  - ❌ "Unleash your operations." / "Scale without limits."
- **Use the trade's words:** kits, load-out / bump-in, prep, dry hire,
  sub-hire / cross-hire, test &amp; tag, road case, dock, FOH, AAA, enquiry.
- **Concrete over vague:** "0 double-bookings this month", "99.9% gear accounted
  for" — never "boost efficiency".
- **Dry wit, lightly.** Real examples in the build: *"built from real production
  work, not imaginary SaaS personas"*, *"priced per warehouse, not per headache"*,
  *"Revolutionary concept: broken things should not be booked."*, *"The show
  still has to go on."*
- **Read it aloud.** Would a production manager say it? If not, rewrite.

**Banned words:** revolutionary*, streamline, seamless, "transform your
workflow", "unleash", "supercharge". (*"revolutionary" only ever as a deliberate
joke, as in the FAQ.) **Never** fake logos or testimonials — frame social proof
as positioning ("Built inside Two Toned Productions · tested on real jobs from
day one"), or use work categories (Dry hire · Full production · Theatre · AV ·
Events) instead of invented company names.

**Product structure / naming.** Modular tiers: **Core** (start here) → **Flow**
(the full connected suite) → **Network** (multi-site / integrations). The product
is organised around the *job*, not inventory.

## 3. Colour

One dominant warm-neutral family + RVLT red + a small functional/categorical set.
**No gradients, ever** (one ≤.12-alpha radial stage-wash excepted, §5). Depth
comes from a luminance ladder, 2px outlines and hard offset shadows.

### 3.1 Dark "Espresso" — PRIMARY (verbatim from `index.html`)

Tint the dark *warm* — never cold slate or pure black (the lesson from Notion
night-shift, Spade, Monologue). The cream keeps continuity with the light theme;
stickers/doodles glow against it.

```css
:root {
  /* surfaces — the luminance ladder (never let two stacked containers match) */
  --paper:#141210; --paper-2:#1A1613; --card:#211C17; --elev:#2A241D;
  /* ink */
  --ink:#F5EFE2; --ink-2:#CDC4B2; --muted:#9E9483; --faint:#6E665A;
  /* lines */
  --line:#332C24; --line-2:#473E32;
  /* brand red (brightened for dark) */
  --red:#E0363D; --red-700:#B21F26; --red-soft:rgba(224,54,61,.16);
  /* categorical (stickers / illustration / charts) */
  --amber:#EBA53A; --blue:#5B8DEF; --green:#3ECF7A; --purple:#9B82E6; --dark:#0E0C0A;
  /* status (text brighter on dark) */
  --ok:#4FD888; --warn:#EBA53A; --rep:#B6AC9A; --t-out:#F26F73;
  --ok-soft:rgba(62,207,122,.14); --warn-soft:rgba(235,165,58,.15);
  --rep-soft:rgba(158,148,131,.14); --out-soft:rgba(224,54,61,.18);
  /* elevation — hard offset (no blur), plus the lit top-edge hairline */
  --sh-card:0 3px 0 #0C0A08; --sh-hover:0 7px 0 #0C0A08;
  --lit:inset 0 1px 0 rgba(255,253,248,.14);
  --sh-stk:2px 4px 0 rgba(0,0,0,.5);
  --r:14px; --r-lg:20px;
}
```

### 3.3 Status palette (functional data — meaning is identical across themes)

| Meaning | Text | Soft fill (dark / light) |
|---|---|---|
| Available / Ready / Pass | `--ok` | `rgba(62,207,122,.14)` / `#E7F5EC` |
| Check / Due / Warning | `--warn` | `rgba(235,165,58,.15)` / `#FBF0D6` |
| Overbooked / Short / Error | `--t-out` | `rgba(224,54,61,.18)` / `#FBE7E3` |
| In repair / Neutral | `--rep` | `rgba(158,148,131,.14)` / `#EFE9DC` |

Status colour encodes *data*, never decoration, and always carries a text label
(never colour-only). A "red row" means the same thing in the mockup and the app.

### 3.4 `.lightui` — the self-lit product UI (dark only)

On the dark page, the real product screenshot is rendered as a **light-mode
island** so it reads as a glowing monitor in a dark room (the #1 pop). Scope a
full light token set onto it:

```css

  --ink:#1D1A15; --ink-2:#4B4539; --muted:#7D7565; --faint:#A89E89;
  --line:#E4DAC3; --line-2:#D8CCB0; --red:#C12229; --red-soft:#FBE7E3;
  --ok:#2EA65C; --warn:#C98A14; --rep:#8A8270; --t-out:#9C1B21;

.shot.lightui{ box-shadow:0 28px 64px -20px rgba(0,0,0,.75), inset 0 1px 0 rgba(255,253,248,.6); }
```

Keep the UI *inside* clean — never put doodles/stickers over live data.

### 3.5 Usage
- **One accent.** Red = the single primary action, active/live state, and alerts.
  If everything's red, nothing is.
- **Name semantically** in code (`--color-action-primary`, `--color-status-overbooked`).
- Surfaces do the work; reach for inverted cream (on dark) only for the spotlight
  pops (§5).

## 4. The spotlight rule — how the dark page pops (the core vibe)

"The page is an **espresso stage in a dark room. Light is a *spotlight*, not the
houselights.**" ~90%+ of every viewport stays in the espresso ramp; brightness is
spent like currency.

- **Light budget = exactly 3 bright "pops" per page**, each a single lit *object*,
  never a section:
  1. the glowing **light-mode product screenshot** in the hero (`.shot.lightui`),
  2. **one cream "spotlight" card** — the featured pricing tier (`.plan.cream`),
  3. **one bounded cream panel** — the testimonial (`.qbox`, contained, dark
     gutters, NOT full-bleed).
  More than ~3, or any light element bigger than a card → cut one.
- **Depth = the luminance ladder, not glow.** `--paper` → `--paper-2` (alt
  sections) → `--card` → `--elev` (one hovered / active / "live" card). Plus 2px
  `--line-2` outline + hard offset `0 3px 0 #0C0A08`.
- **Lit top-edge hairline** (`--lit`) is reserved for `--elev` surfaces only —
  sticky nav, the one "live" card, hover/active. It signals hierarchy; it is not
  decoration.
- **One warm radial stage-wash** behind the hero only — amber
  `radial-gradient(closest-side, rgba(235,165,58,.11), transparent 72%)`, `aria-hidden`.
  The single justified glow; "a light, not a blob."
- **Red, tiered to one moment per viewport:** hero keyword in red + scribble →
  `--red-soft` "now" ring → status colour quarantined *inside* the product UI →
  a single haloed CTA (`.btn-halo`) → the full-red closing CTA block as the
  crescendo. `#E0363D` on dark (never `#C12229` — it vibrates).
- **Rest valleys.** Keep the logos &amp; integrations strips deliberately quiet so
  the pops land. Frosted glass = sticky nav only.
- This **replaces full light/dark banding** (an earlier rejected experiment):
  modulate *within* the espresso ramp for
  section contrast — never flip the whole page to day.

## 5. Typography

Four families. **Display = personality. Body = quiet. Handwriting = the human
touch. Baloo 2 = the wordmark only.**

| Role | Family | Weights | Notes |
|---|---|---|---|
| Display | **BC Alphapipe** (Adobe Fonts) | 700 / 800 / 900 | Logo mark + all headlines + step numbers + stat figures. Pipe-cut terminals = the fingerprint. |
| Display fallback | **Archivo** (Google) | 700 / 800 / 900 | Holds layout until the Adobe kit loads. |
| Body | **Hanken Grotesk** (Google) | 400 / 500 / 600 | Paragraphs, UI, nav, labels. Readable — *not* Inter. |
| Handwriting | **Kalam** (Google) | 400 / 700 | Eyebrows/kickers + doodle annotations + the cream-tier note. Playful **and legible** (replaced Caveat, which was too hard to read). Often red, slight rotation. |
| Wordmark "Flow" | **Baloo 2** (Google) | 700 | The "Flow" half of the lockup *only* — rounded counterweight to the pipe-cut mark. Not a headline face. |

Wire display through a CSS var so the Adobe swap is one line:
```css
--fd: "bc-alphapipe", "Archivo", sans-serif;
```
**BC Alphapipe** is web-licensed via the RVLT Adobe subscription — embed an Adobe
**Web Project** kit (`<link rel="stylesheet" href="https://use.typekit.net/XXXXXXX.css">`,
then `font-family:"bc-alphapipe"`). Kit URL is account-specific. *(Still pending.)*

### 5.2 Rules
- **Sentence case** for everything readable. No ALL-CAPS shouting (that was the
  rejected industrial misfire). The eyebrows are **handwritten Kalam, sentence
  case** — playful but legible, e.g. *"why Flow exists"*, *"built from the actual
  job flow"*.
- **High-contrast jumps** — huge tight display against small quiet body; one
  bright stat figure (`38px --ink`) while its siblings drop to `--ink-2`.
- Keep handwriting for *eyebrows + doodles only* — never body or long text.

### 5.3 Logo lockup pairing
```css
.brand .fl{ font-family:"Baloo 2",cursive; font-weight:700; letter-spacing:-.01em; }
```
RVLT = BC Alphapipe SVG mark (pixel-accurate). Flow = Baloo 2. Baloo 2 is
reserved for the lockup.

### 5.4 Product app: a data mono
For dense screens add a tabular monospace for asset IDs, quantities, timestamps,
SKUs: **Geist Mono** or **JetBrains Mono**, `font-variant-numeric:tabular-nums`.
Data cells only — not body, not the marketing site.

## 7. Shape, borders &amp; elevation

Signature look = **2px outline + hard offset shadow (no blur)** — tactile, like
printed flight cases and enamel patches; the opposite of soft-blur SaaS.

```css
--r:14px;        /* default cards, faq */
--r-lg:20px;     /* large cards, showcases, hero shot */
/* pills/badges/buttons/chiplogos: 99px */
--sh-card:0 3px 0 #0C0A08;   /* dark; light = 0 3px 0 var(--line-2) */
--sh-hover:0 7px 0 #0C0A08;  /* + translateY(-3px) */
--lit:inset 0 1px 0 rgba(255,253,248,.14);  /* lit edge — --elev surfaces only */
--sh-stk:2px 4px 0 rgba(0,0,0,.5);          /* sticker/patch (light: rgba(40,30,15,.13)) */
```
- **Card border:** `2px solid` — `--line-2` on dark, `--ink` on light.
- **Inner rows / hairlines:** `1px solid var(--line)`.
- **Only soft shadows allowed:** the `.shot.lightui` float (§3.4) and the cream
  pop panels. Everywhere else, hard offset only.

## 8. Motion

Weighty, physical, purposeful — never ambient decoration. Always honour
`prefers-reduced-motion` (disable loops + parallax, freeze at rest).

- **Button press:** primary lifts 1px on hover, drops 2px on `:active`
  (shadow shrinks to `0 1px 0`). Tactile.
- **Hero CTA halo:** `.btn-halo` adds a soft red bloom to the *one* primary hero
  CTA — the only place red glows.
- **Signature loops (subtle, ≤2):** live dot `pulse 1.7s`; pipeline segment
  `grow 2.4s ease-in-out`.
- **Hero sticker parallax (the one scroll effect).** *Only* the hero gear
  stickers (`.hero .stk`) drift on scroll — doodles, the dashed circle, and CTA
  stickers stay put. Implementation: rAF + `passive` scroll; per-sticker varied
  amplitude (≈44–116px) and interleaved direction so they scatter, not march;
  `translate3d(0,y,0)` composed *before* each sticker's rotation so tilt is
  preserved; recompute mids on resize; bail entirely under reduced-motion.

## 10. Illustration &amp; sticker system

The collage of **die-cut gear stickers** is the most ownable element — road cases
are literally covered in tour stickers, so it's product-true, and the cream
die-cut borders glow on the espresso canvas.

**Sticker (patch) recipe:** a bold shape (rounded `16px` square or circle) in a
categorical colour · `3px` cream (`#FFFDF8`) border = the die-cut edge ·
`--sh-stk` offset · varied rotation `±6–11°` · a white `~2.6px`-stroke icon, or a
short display word (`GAFF`, `AAA`).

**Gear vocabulary** (recurring cast): road/flight case · PAR can · moving head
(= the mascot) · XLR connector · gaff-tape roll · "AAA" access pass · cable loom ·
wrench · fader/console. Flat, bold, 2–3 colours each.

**Hand-drawn doodles** (the PostHog / Cloudflare sketchbook energy) — **Kalam**,
red, slight rotation: scribble underline beneath a key word; dashed ellipse
circling something; a hand-drawn arrow + short note (e.g. *"caught before the
truck leaves"*, *"one change, everywhere"*).

**Rules:** stickers/doodles cluster around the hero and CTA — **never inside
data-dense product screens**. Personality lives in the framing, not on the data.
The current SVGs are concept sketches; a proper illustrator pass is a known next
step.

## 12. Anti-slop rules (never ship these)
- ❌ Blue→purple / purple→pink gradients (the Tailwind `indigo-500` tell); gradient
  text; glow blobs; soft ambient drop-shadow everywhere.
- ❌ Inter / Roboto / system-ui as the *display* face.
- ❌ The generic 3-icon-cards row with abstract glyphs.
- ❌ Uniform 16px-radius / 24px-padding flatness with no hierarchy.
- ❌ Vague aspirational headlines; banned SaaS words (§2).
- ❌ 3D abstract blobs, AI stock photos, "smooth plastic" illustration, fake
  logos/testimonials.
- ❌ Over-correcting into heavy-industrial cosplay (hazard tape everywhere,
  all-caps mono, barcodes as wallpaper) **or** sterile minimal-SaaS. Character ≠
  severity *and* ≠ blandness — keep it warm, witty, legible.
- ❌ Cold-slate or pure-black dark mode; light sections that take a whole screen
  (breaks the spotlight rule).

**Ship checklist:** zero gradients (one ≤.12 stage-wash ok) · BC Alphapipe/Archivo
display · one red accent, tiered · ≤3 light pops on dark · luminance-ladder depth
+ 2px outline + offset shadow · operator-voice headline, no banned words, no fake
proof · real icon set · Kalam handwriting legible · motion only on state change +
parallax on hero stickers only + reduced-motion · gear personality in the framing,
product data clean · squint test: crew-made, not template?

## 13. Accessibility
- Body ≥ 16px in product; nothing below 11px anywhere.
- WCAG AA contrast — status text uses the darker stop on light (`#9C1B21` etc.)
  and the brighter stop on dark (`--t-out #F26F73`). Verify red-on-cream.
- Never encode state by colour alone — badges always carry a text label.
- Honour `prefers-reduced-motion` (loops + sticker parallax off).
- Stickers/doodles are decorative → `aria-hidden`; the stage-wash glow too. The
  mascot needs alt text only when it carries meaning (empty-state messages).

## 15. Building the app (authoritative for app code)

Stack: Next.js 16 · TS · **Tailwind v4** · **shadcn/ui** · Postgres/Prisma.
Dark-first: `:root` = espresso, `.light` = alternate.

**The Functional dial = "calm data, fun chrome."** Keep colour tokens, type
families, status palette, red-for-primary, the 2px-outline + offset-shadow
character. Dense data screens stay clean (no stickers/doodles on tables; data
mono for figures; tighter spacing). **All the energy goes into the chrome and the
moments** — sidebar, headers, modals, toasts, empty states, loading, success
(§15.5). Build the signature screens per §15.4. Motion = press feedback + state
transitions + the sanctioned moment animations; no scroll parallax in-app.

### 15.2 Component recipes (shadcn, restyled to RVLT)
- **Button / primary:** `bg-red text-white rounded-full font-semibold shadow-[0_3px_0_var(--red-700)] hover:-translate-y-px active:translate-y-[2px] active:shadow-[0_1px_0_var(--red-700)] transition`.
  - `outline` = `border-2 border-line-2 bg-transparent hover:bg-ink hover:text-paper`; `ghost` = text only, `hover:text-red`.
- **Card:** `bg-card border-2 border-line-2 rounded-[var(--radius)] shadow-[var(--sh-card)]` (override shadcn's soft shadow). Active/selected/"live" → `bg-elev shadow-[var(--sh-card),var(--lit)]`.
- **Badge (status):** `rounded-full text-[11px] font-bold px-2.5 py-1` + a status pair — ready `text-ok bg-[var(--ok-soft)]`, check `text-warn bg-[var(--warn-soft)]`, overbooked `text-t-out bg-[var(--out-soft)]`, repair `text-rep bg-[var(--rep-soft)]`. **Always render the label** (never colour-only).
- **Input / Select / Textarea:** `bg-card border-2 border-line-2 rounded-[var(--radius)] focus-visible:ring-2 focus-visible:ring-red`.
- **Table / data:** numeric columns `font-mono tabular-nums text-ink-2`; status cell = Badge; row divider `border-t border-line`.
- **Type:** headings `font-display font-extrabold tracking-tight`; section eyebrow `font-hand text-red text-[17px]`; body `font-sans`; figures `font-mono tabular-nums`.
- **Empty state:** the Flow mascot + one operator-voice line — the one sanctioned place for personality in-app.

### 15.4 Signature product-UI patterns (build the app to look like these)

These three mockup segments are the look the founder wants the app to *be*. They
double as real app components; markup lives in `index.html`.

**(a) Availability board / project view** — *the glowing hero `.lightui` dashboard.*
The hero-status of the whole product: a project with its lifecycle and live gear
status. Anatomy:
- **Frame:** a `--card` panel, 2px border, `--r-lg`, window-style header (project
  title + breadcrumb + a live status pill). In-app it's the page surface; on the
  marketing site it's rendered light-on-dark to glow (§3.4).
- **Lifecycle stepper (the spine):** horizontal stages Enquiry → Quote → Confirmed
  → Prep → On site → Return; each = a 28px ring + label. States: **done** =
  `--ink` fill + check · **now** = red ring + `--red-soft` glow ring + bold label ·
  **future** = `--line-2` ring, `--faint` label. Connectors: completed = `--ink`;
  the *active* segment animates (`--red` `scaleX`, `grow 2.4s`). This is the
  canonical project-status component — rings clickable to jump stage.
- **"Live" pill:** green soft-fill + pulsing dot = real-time / on-the-floor.
- **Data rows (availability):** grid `1fr auto auto` = asset (name + small meta) ·
  qty (`font-mono tabular-nums`) · **status badge** (Ready `--ok` · Check `--warn`
  · Overbooked/Short `--t-out` · In repair `--rep`). Row hover → `--elev`. **This
  table stays clean — the colour IS the fun; never decorate the data.**

**(b) "What Flow connects" — mini-UI feature cards → the dashboard widgets.**
Each card = a coloured **patch icon** + title + a ~118px **live mini-UI** + one
line of copy. In-app these mini-UIs are the *real* dashboard widgets (scan feed,
availability bars, quote doc, crew strip, compliance list, repair queue) — build
the full-size widget from the same parts and tokens.
- **Patch icon:** 42px rounded square, a **module-colour** fill, 2px cream border,
  `--sh-stk`, white Lucide glyph. The module colour is the app's wayfinding system
  — the same hue appears on the sidebar item, the page header and this icon (map
  in §15.5).
- **Mini-UI tile:** on `--paper` inside the `--card`, bordered top+bottom by
  `--line`; a miniature of the real widget built from the same tokens (chips,
  `--mbar` bars in status colours, a doc, a gantt).
- **The one "live" card:** lifts to `--elev` + `--lit` edge + a pulsing "live" tag
  — the pattern for any real-time widget on the dashboard.

**(c) Crew scheduler** — *the "why Flow" timeline → the real Crew screen.*
A 7shifts/Deputy-style timeline: a day/hour column header over per-person rows.
Each row = an **avatar** (initials on a per-person colour) + name + a track
holding a **shift bar** placed by `left%`/`width%` = a booking, white label on the
person/job colour. An **"+N open"** row shows unfilled calls.
- **Colour logic:** every crew member / job gets a categorical colour so the board
  is scannable at a glance — the antidote to a grey roster. **Clashes** (overlaps)
  flag in overbooked red with a callout.
- Drag-to-assign; on drop, Flow checks availability, skills &amp; tickets. One hand
  "no clashes ✦" note in the corner is fine here (chrome, not data).

### 15.5 Make the app FUN — "calm data, fun chrome"

> Production software is black, grey and boring. RVLT Flow is **warm, coloured and
> has a pulse.** The rule that lets us be fun *without* hurting usability: **dense
> data stays calm and legible; energy goes into the chrome and the moments around
> it** — sidebar, headers, modals, toasts, empty states, loading, success.
> Colour-code everything; give each surface exactly one personality beat.

**Module colour map** (wayfinding + joy — one hue per area, on its sidebar icon,
page-header accent and feature patch). Red is reserved for primary-action /
active / alert, so modules draw from the categorical ramp:

| Module | Colour |
|---|---|
| Projects / Jobs | `--blue` |
| Gear / Inventory | `--amber` |
| Crew / People | `--purple` |
| Availability | `--green` |
| Compliance | `--green` (tint) / add `--teal` |
| Maintenance | `--rep` neutral / add `--coral` |

Add `--coral` / `--teal` to the ramp if you need more distinct module hues. Keep
status badges as **labelled soft-fill pills** so a green *badge* never reads as the
green *module*.

**Sidebar** — warm espresso (`--paper-2` / `--card`), **never black**. Grouped nav,
Lucide icons tinted by module colour; the active item = `--elev` bg + `--lit` top
edge + a coloured/red left indicator pill. Tuck the **Flow mascot** at the bottom
(idle); collapses to an icon rail. Rounded, tactile, a little alive.

**Modals / dialogs** — an `--elev` (or cream) panel: 2px outline + `--sh-card` +
`--lit` edge, bold BC Alphapipe title, chunky pill buttons (red primary, tactile
press). One small sticker/doodle accent in the header corner is allowed (sparingly).
Copy is human even when destructive: *"This kit's booked on Saturday — pull it off
that job anyway?"*

**Toasts / notifications** — chunky card/pill: status colour + Lucide icon + a bit
of wit, `--sh-card` offset, quick pop-in (`translateY` settle ~200ms,
reduced-motion aware). Scan-out success → green toast *"Checked out · K2 #04 ✓"*
with a tiny stamp. A whole job returned clean → a **mascot cameo** + a few flat
confetti shapes (tasteful, motion-safe). Errors are red but never robotic.

**Empty states** — the **mascot** + one operator-voice line: *"Nothing booked yet
— Flow's having a quiet one."* / *"No clashes. Suspiciously calm."* The sanctioned
home for personality in the app.

**Loading** — skeletons in the luminance ladder (`--card` / `--elev`); the mascot
in "scanning" pose (beam on) for longer waits.

**Micro-moments (one per surface, never everywhere):** the button press drop; the
lifecycle segment growing red; a status badge cross-fading on change; the "now"
ring pulse; a row settling after a scan. Always honour `prefers-reduced-motion`.

**Through-line:** warm surfaces + the categorical ramp + one personality beat per
screen = energy. Keep it off the dense data; put it everywhere around it.
